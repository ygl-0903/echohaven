# 余音存储设计说明：加密金库 vs SQLite

本文用通俗说法说明：**余音里的「vault」是什么**、**和 SQLite 有什么不一样**、**为什么当前实现选单文件加密金库**，以及各自优缺点。实现细节见 [vault-format-v1.md](vault-format-v1.md)。

---

## 「vault」到底是什么？

在余音里，**vault 不是像 SQLite 那样的国际标准格式名**，而是**我们给「一整包加密数据文件」起的名字**（行业里密码管理器也常用 vault 表示「金库」）。

磁盘上的表现是：**一个 `.vault` 文件** = 文件头（魔数、版本、KDF 盐与参数）+ **一整块经过认证加密的内容**。解密后，当前实现里是一份 **JSON**（条目、提醒等），**不会在磁盘上留下明文数据库文件**。

所以：你之前「没听说过某一种叫 vault 的通用格式」是正常的——**它是应用自定义的容器格式**，规范以本仓库 `docs/vault-format-v1.md` 与 `src-tauri/src/vault.rs` 为准。

---

## 和 SQLite 有啥区别？

| 维度 | 余音 `.vault`（单文件加密金库） | SQLite（`.sqlite` / `.db`） |
|------|----------------------------------|-----------------------------|
| **是什么** | 自定义的**加密容器**：外面是固定头，里面是**密文** | 公开的**嵌入式数据库引擎**，文件内部是结构化的页、表、索引 |
| **磁盘上** | 看不到明文条目；改一条也要重写加密块（当前实现为整库 JSON） | 明文结构可被专用工具直接打开（除非你再套一层加密，如 SQLCipher） |
| **查询方式** | 先解密，再在内存里处理；**没有 SQL** | **SQL** 查询、多表、索引、事务 |
| **备份/迁移** | **复制一个文件**即带走整库，适合「手动备份 U 盘」 | 也是一个文件，但**未额外加密时**备份的是**可被 SQLite 工具读取**的数据 |
| **生态** | 只有按我们格式实现的程序能解锁 | 工具、语言绑定、浏览器插件极多 |

一句话：**SQLite 解决的是「结构化怎么存、怎么查」；余音的 vault 解决的是「整份敏感数据怎么加密成一包、怎么备份」**。

---

## 哪种更合适？（按目标选）

没有绝对「更好」，要看产品目标：

- **更适合单文件加密金库（当前余音路线）**  
  - 强调：**离线、无服务端、备份就是拷贝一个文件**  
  - 威胁模型：**磁盘上的文件被拷走也不能直接读**（需主密码 + KDF）  
  - MVP：**实现路径短**——Rust 里 Argon2 + AEAD 包一层序列化数据即可  

- **更适合 SQLite**  
  - 条目**非常多**、需要复杂筛选/排序/全文索引且希望**不解密整库**也能做部分优化（通常要结合分页加密或字段级加密，设计会更复杂）  
  - 需要**多进程并发写**、复杂迁移与关系型建模  
  - 若仍要「文件被拷走也安全」，一般要 **SQLCipher** 或应用层加密，已不是「裸 SQLite」  

余音规划里是**单设备、手动备份、金库文件防拷贝直读**，因此 **MVP 选单文件加密金库更合适**。

## 当前方案的优点与代价

**优点**

- **语义简单**：用户理解「一个加密文件 = 我的全部密码库」。  
- **备份/恢复简单**：复制、替换文件；与「不做云同步」一致。  
- **安全边界清晰**：敏感内容只在**解密后的内存**里以结构化形式存在（仍受操作系统与恶意软件制约，见产品威胁模型）。  
- **依赖少**：不必嵌入 SQLite 引擎；密码学集中在 Rust，便于审计与测试。

**代价 / 注意点**

- **大库性能**：当前是「整包解密 / 加密保存」，条目数极大时 CPU 与写入会上升；缓解方向包括分页加密、二进制增量格式、或引入带加密扩展的存储（工作量和兼容性都会增加）。  
- **不是通用格式**：其他软件不会自动识别 `.vault`，除非对方实现同一规范。
